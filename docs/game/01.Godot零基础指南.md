# Godot 像素画模糊原理 + Nearest 过滤完整解释
## 一、核心根源：纹理采样默认是线性过滤 Linear
游戏引擎渲染图片时，**放大/缩小图片**，显卡要做插值计算填充中间像素，有两种采样模式：
1. **Linear（线性过滤，Godot默认）**
   取相邻4个像素做平滑混合，放大后边缘柔和、模糊，适合照片、插画、3D贴图。
2. **Nearest（邻近过滤/最近点采样）**
   直接取离采样点最近的那一个像素，不混合，硬边缘、清晰方块，专门给像素画用。

### 直观例子
一张16×16像素小人，放大3倍到48×48：
- Linear：像素之间渐变过渡，出现模糊灰边、锯齿糊块
- Nearest：每个原像素直接变成3×3方块，边界锐利，纯像素风格

## 二、为什么只改Sprite没用，必须改纹理导入设置？
Godot的纹理参数**存在资源文件本身**，不是Sprite节点属性：
1. Sprite2D/TextureRect 只是“显示图片的容器”，本身没有像素采样开关；
2. 图片导入后生成 `.import` 配置文件，过滤模式、mipmap、重复都写在这里；
3. 哪怕节点缩放、摄像机拉远拉近，采样规则都由纹理资源决定。

### 正确设置步骤（Godot 4 / Godot 3通用）
1. 选中你的png像素素材，右侧打开「Import 导入面板」
2. **Filter → Nearest**（关键）
3. 取消勾选 `Mipmaps`（Mipmap多层缩小贴图会额外糊像素画）
4. 点击底部「Reimport 重新导入」
5. 此时场景里所有使用这张图的Sprite都会变清晰

## 三、额外会让像素画变糊的隐藏坑
### 1. 摄像机/节点缩放不是整数倍
比如原图16px，缩放 2.3 倍，不是整数倍数，哪怕Nearest也会轻微错位模糊。
解决：缩放统一用整数 `1 / 2 / 3 / 4`。

### 2. 窗口分辨率、相机大小不匹配像素尺寸
像素游戏标准做法：
- 设定原生像素分辨率（如320×180）
- Camera2D 开启 `Smoothing = false`，防止相机亚像素偏移
- Project设置 → Display → Window 拉伸模式选 `Viewport`，拉伸过滤同样设Nearest

### 3. 导入开启了 Compress 纹理压缩
压缩算法会破坏纯色像素边界，像素画建议：
Import面板 → Compression → `Lossless Lossless（无损压缩）`，不要VRAM压缩。

### 4. 节点位置是小数坐标
Sprite位置 `x=10.5` 这种小数，会导致像素偏移糊边。
代码/编辑器强制坐标为整数：`position = position.round()`

## 四、底层显卡原理补充
显卡渲染时纹理坐标是浮点数：
假设原图像素坐标 `0~1` 代表第一个像素，`1~2` 第二个。
- Linear：浮点数落在两个像素中间，取两者颜色混合
- Nearest：直接四舍五入取整数像素，无混合，完美保留像素方块感

## 五、快速总结
像素画模糊本质是**默认线性插值平滑像素**；
把纹理过滤改成 Nearest = 关闭平滑插值，直接取原始像素，画面就不会发糊。
且该配置绑定纹理资源，只改Sprite节点无法生效，必须重新导入纹理。

## velocity 是什么？

## 1. 字面含义
**velocity = 速度、速率**，物理里指**矢量速度**（带方向+大小，区别于单纯速度大小 speed）。
- speed：只有快慢（标量）
- velocity：快慢 + 移动方向（矢量 Vector2 / Vector3）

## 2. 在 Godot 里它是什么对象？
`velocity` 不是内置固定类，**它就是一个矢量变量**：
- 2D游戏：`Vector2` 对象（x 水平速度，y 垂直速度）
- 3D游戏：`Vector3` 对象（x/y/z三轴速度）

### Vector2 矢量对象自带属性
```gdscript
var velocity: Vector2
velocity.x = 100   # 水平向右速度
velocity.y = -200  # 向上速度
velocity.length()  # 获取移动总快慢（等价speed）
velocity.normalized() # 单位方向向量
```

## 3. 谁自带 velocity 属性？（常用节点）
### CharacterBody2D / CharacterBody3D（角色移动核心）
引擎内置自带 `velocity` 成员变量，类型就是 Vector2/Vector3：
```gdscript
# 角色移动标准写法
func _physics_process(delta):
    var input_dir = Input.get_vector("left", "right", "up", "down")
    velocity = input_dir * move_speed
    move_and_slide() # 内部自动读取velocity移动
```
这里的 `velocity` 是节点自带的**Vector2 矢量对象**，专门存储这一帧要移动的速度。

### RigidBody2D 刚体
刚体不用 velocity，用 `linear_velocity`（同样 Vector2，线性速度矢量）。

## 4. 核心作用
1. 保存物体本帧移动的方向与快慢；
2. 交给物理函数（move_and_slide）计算位移；
3. 重力、跳跃、摩擦力都是直接修改 velocity 的 x/y 值实现。

## 5. 举个直观例子
`velocity = Vector2(150, 0)`
代表物体**每秒向右移动150像素**；
`velocity = Vector2(0, -300)`
代表向上冲刺，y负为Godot2D向上。

## 简短总结
velocity 本质是 **Vector2/Vector3 矢量对象**，专门存放带方向的移动速度，角色体节点内置这个变量用来驱动物理移动。

**move_and_slide() 这个函数的作用？**

# move_and_slide() 完整讲解（Godot CharacterBody2D 专用）
## 一、基础作用
`move_and_slide()` 是 **CharacterBody2D / CharacterBody3D** 自带的物理移动函数，
读取节点自身的 `velocity` 矢量，完成**带碰撞检测的平滑移动**，自动处理碰撞、墙面滑行、地面判定、台阶等逻辑。

## 二、核心干了哪些事（一步一步）
1. **读取 `velocity`**
   拿你代码里赋值好的 `self.velocity` 作为移动向量。
2. **按物理帧时间自动换算位移**
   不需要你手动乘 `delta`！内部已经处理好物理时间步，直接填速度数值即可。
3. **逐段移动 + 碰撞检测**
   如果移动路径撞上碰撞体，不会直接穿墙，会停下并修正速度。
4. **碰撞自动分离 & 滑行**
   - 撞墙：水平速度被抵消，不会贴墙卡住抖动；
   - 撞斜面：自动沿坡面滑动，不用自己写斜面逻辑；
5. **更新地面/墙壁/天花板状态**
   执行后才能用这几个判断：
   - `is_on_floor()` 是否踩在地上（跳跃、重力判定核心）
   - `is_on_wall()` 是否贴墙
   - `is_on_ceiling()` 是否头顶撞顶
6. **返回本次移动后修正完的 velocity**（可选接收）
```gdscript
velocity = move_and_slide()
```

## 三、和普通 `move_and_collide()` 的区别
1. `move_and_slide()`
   专为玩家角色设计，自动处理斜面、地面、摩擦力、滑行，配合内置 `velocity` 使用，写平台跳跃、横版角色首选。
2. `move_and_collide()`
   底层通用移动，只单纯阻挡碰撞，不会修正速度、不识别地面斜面，一般子弹、道具用。

## 四、最简单标准模板（2D角色）
```gdscript
extends CharacterBody2D
@export var speed = 300
@export var jump_power = 450
@export var gravity = 1800

func _physics_process(delta):
    # 水平输入
    var input = Input.get_vector("left", "right")
    velocity.x = input * speed

    # 重力
    if not is_on_floor():
        velocity.y += gravity * delta
    # 跳跃
    if Input.is_action_just_pressed("jump") and is_on_floor():
        velocity.y = -jump_power

    # 核心移动函数，自动读取velocity并修正
    velocity = move_and_slide()
```

## 五、关键注意点
1. **必须写在 `_physics_process(delta)` 里**
   物理逻辑固定60帧运行，不能放 `_process`，否则移动快慢不稳定。
2. **不用手动 velocity *= delta**
   函数内部自动处理时间缩放，手动乘会速度翻倍变快。
3. 执行完 `move_and_slide()` 后，`is_on_floor()` 才有效
   先移动，再判断是否落地，顺序不能颠倒。
4. 斜面滑行自动处理
   上坡减速、下坡加速、贴墙滑，不用自己计算法线。

## 简短总结
`move_and_slide()` 读取角色体的 `velocity`，执行带碰撞、斜面、地面检测的角色移动，自动修正碰撞后的速度，是2D/3D玩家角色移动的核心API。

**Godot _ready() 函数完整详解：**

当当前节点 + 它所有子节点全部加载、实例化完毕后，自动执行一次，只运行 1 次。

执行顺序：场景实例化 → 创建所有节点、赋值@onready变量 → 调用每个节点的 _ready() → 之后才会跑 _process / _physics_process

## 瓦片地图

# TileSet 与 TileMap（Godot4 含 TileMapLayer）完整关系拆解
## 一、通俗比喻（一秒分清）
- **TileSet（瓦片集）= 积木素材盒（资源 Resource）**
  存放所有瓦片图片、碰撞形状、物理层、自动瓦片、导航、动画等**瓦片本身的固定属性**，是独立素材资源，可以被多个地图复用。
- **TileMap / TileMapLayer（瓦片地图层）= 拼积木的底板（场景节点 Node）**
  只是一块空白网格画布，**自身不存任何瓦片素材**，只会引用一份 TileSet，记录「哪个格子放哪块瓦片」的摆放坐标。

简单一句话：**TileSet 管瓦片长啥样、有碰撞；TileMapLayer 管瓦片摆在哪、怎么分层渲染**。



## 二、核心本质区别
### 1. TileSet：资源（Resource），独立文件
- 类型：`TileSet` 资源，可保存为 `.tres` 文件放在项目资源文件夹；
- 存储内容：
  1. 图集贴图、瓦片尺寸、分割规则
  2. 所有瓦片的**碰撞多边形、物理层、单向平台**（你之前编辑物理层就是改它）
  3. 自动瓦片（Autotile）、瓦片动画、导航区域、遮挡信息
- 复用性：**一张 TileSet 可以给无数个 TileMapLayer 使用**，改一处全部地图同步更新；
- 关键限制：碰撞、贴图、瓦片外形全部锁死在 TileSet，地图节点无法修改瓦片本身属性。

### 2. TileMapLayer（Godot4 新版推荐节点）：场景画布节点
Godot4 拆分了旧版 TileMap，现在标准用法是单独 `TileMapLayer` 节点：
- 类型：场景树里的 Node2D 子节点，只存在当前场景；
- 只存两类数据：
  1. 引用一个 TileSet 素材盒；
  2. 网格坐标记录：`(0,0)放草地、(1,0)放墙` 这类摆放信息；
- 自身属性：图层显隐、Y排序、渲染层级、物理区块大小、偏移、透明度；
- 分层逻辑：一张地图做多图层（地面层、墙体层、装饰层），每个图层都可以绑定同一个 TileSet。

### 3. 旧版 TileMap（兼容保留，不推荐）
一个 TileMap 内部包含多个内置图层，逻辑和多个 TileMapLayer 一致，只是封装在一个节点里，新版开发优先用独立 TileMapLayer。

## 三、二者绑定关系
1. 新建 `TileMapLayer`，右侧检视器 `Tile Set` 属性为空；
2. 新建/加载外部 `.tres` TileSet 资源完成绑定；
3. 底部编辑器读取 TileSet 里所有瓦片，提供画笔供你在画布绘制；
4. 绘制时：你只是**从 TileSet 里取出瓦片放到图层网格**，瓦片的碰撞、贴图完全由 TileSet 决定。

## 四、结合你之前遇到的物理层问题解释
1. 为什么物理层只能单块瓦片编辑？
   碰撞形状**存在 TileSet 资源内部**，TileSet 以「单瓦片格子」为最小单位存储碰撞，地图图层只负责摆放，不能修改瓦片物理形状。
2. 为什么改完碰撞，所有使用该 TileSet 的地图都生效？
   所有图层共用同一份 TileSet 资源，资源统一管理瓦片属性。
3. 图层不能单独修改某块瓦片碰撞？
   不行，瓦片属性全局绑定 TileSet；想要特殊碰撞瓦片，必须在 TileSet 里新增一块独立瓦片。

## 五、典型工作流程
1. 导入瓦片图集PNG，新建 TileSet，设置瓦片尺寸、分割图集；
2. 在 TileSet 编辑器添加物理层，给墙体/地面绘制碰撞框；
3. 保存 TileSet 为外部 `.tres` 文件（方便多场景复用）；
4. 场景创建多个 TileMapLayer（地面、墙体、前景装饰）；
5. 所有图层统一绑定刚才做好的 TileSet；
6. 在各个图层画笔绘制地图，自动携带碰撞。

## 六、关键总结
1. **TileSet = 瓦片素材库（管贴图、碰撞、物理），可全局复用**
2. **TileMapLayer = 场景绘制画布（只管瓦片摆放位置、分层渲染）**
3. 依赖关系：TileMapLayer **必须依赖 TileSet 才能绘制地图**，反过来 TileSet 可以脱离地图独立存在；
4. 瓦片物理、外形、动画统一在 TileSet 修改，地图图层仅负责布局。

## Y轴排序

# 问题根源
Y排序是**对比物体底部Y坐标**决定遮挡，你现在人物的**排序基准点（脚底）和精灵画面底部没对齐**，引擎只看脚底位置，忽略头顶高度，走到树下时脚底Y比树干低，就会让树画在人物下方，头顶穿出树叶。

## 分三步彻底修复（对应你这张俯视像素图）
## 1. 修正角色自身：把Y排序基准压到人物脚底
你的角色是`CharacterBody2D` + 子节点`AnimatedSprite2D`
1. 选中根节点Player（CharacterBody2D）
   - `Z Index = 0`，不要开自身`Y Sort Enabled`
2. 选中内部`AnimatedSprite2D`
   - 修改 `Position Y` 向上偏移（负数），让**精灵图片的底边刚好贴住Player原点**
   举例：人物整张图高32像素，就设置 `sprite.position.y = -32`
   效果：引擎排序计算时，拿人物脚底（原点）做对比，而不是精灵中间/头顶。

## 2. 树木TileMapLayer图层关键设置
选中放树木、草丛的TileMapLayer：
1. `Z Index = 0`（和人物统一）
2. **Y Sort Origin = 瓦片完整高度**
   比如你的树瓦片是48px高，就填`48`
   含义：瓦片的排序基准点在**瓦片格子最底边**（树根位置），引擎用树根Y和人物脚底Y对比。

## 3. TileSet里修正树木贴图偏移（最关键一步，90%人漏这个）
1. 打开底部TileSet编辑器，切换`Select`模式，选中大树瓦片
2. 右侧找到 `Texture Offset Y`，填负数向上偏移贴图
   目标：让**树叶贴图的底边对齐瓦片格子的底边**
   默认贴图原点在格子中心，树叶下半截悬空，偏移后树根死死贴格子底部，Y坐标基准才正确。

### 原理演示
- 没偏移：树贴图在格子中间，树根Y坐标偏高 → 人物脚底Y < 树根Y → 人物画在树上面，头顶穿树叶
- 偏移对齐底部：树根贴格子底边，树根Y坐标更低 → 人物走到树下时脚底Y > 树根Y → 树完整盖住人物上半身

## 4. 父容器总开关检查
场景顶层父节点（比如World/Map）
- `Y Sort Enabled = true`
- `Z as Relative = true`
所有图层、角色都作为它的直接子节点，不能嵌套多层父节点。

## 补充避坑方案（高大植被专用）
如果调整完还是轻微穿模，可以拆分两层地图：
1. **下层：地面、树干层** Z=0，参与Y排序
2. **上层：树叶树冠层** Z=1，不开启Y排序，永久置顶
人物永远被上层树叶盖住，完美解决头顶穿出问题。

## 快速自检清单
1. 父节点开全局Y Sort，所有物体Z=0 ✅
2. 角色精灵向上偏移，脚底对齐角色原点 ✅
3. 树木图层Y Sort Origin等于瓦片高度 ✅
4. TileSet内树木贴图上移，树根贴瓦片格子底边 ✅
